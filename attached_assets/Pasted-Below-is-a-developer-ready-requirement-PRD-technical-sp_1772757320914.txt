Below is a developer-ready requirement (PRD + technical specs) for a University Agreement & Commission Management Portal with high security, role-based access, secure in-portal document viewing (no download), expiry reminders, and React (frontend) + Python (backend).

1) Goal

Build a secure internal portal to manage University Agreements and all related details:

Agreement metadata: start/end/expiry, territory, commission rules, targets

Upload agreement files (PDF/DOC) with strict permissions

View-only mode must allow in-portal review without download

Multiple university contacts for renewal/updates

Role-based management (some users can add/edit, some can only view)

Automated expiry reminders to:

au@studyinfocentre.com

info@studyinfocentre.com

partners@studyinfocentre.com

2) Users & Roles
2.1 Role Types (minimum)

Super Admin

Full access to everything

Manage roles/permissions

Agreement Admin

Create/Edit agreements, targets, commission rules

Upload/replace documents

Manage contacts

Agreement Editor

Edit agreement details (but cannot manage roles)

Upload documents (optional toggle)

Agreement Viewer (High security)

Can view agreement details

Can view documents only inside portal (no download)

Document Viewer Only

Can only view document viewer, no metadata edits

Auditor/Compliance

Read-only + access to audit logs

Roles must be assignable per user, and optionally per territory/country.

3) Permission Model (Fine-grained)
3.1 Agreement permissions (per agreement or per university)

agreement.view

agreement.create

agreement.edit

agreement.delete (optional)

agreement.export (optional)

agreement.targets.manage

agreement.commission.manage

agreement.contacts.manage

3.2 Document permissions

document.upload

document.replace

document.view_in_portal ✅

document.download ✅ (only for permitted users)

document.delete (optional)

3.3 Special enforcement

If user has view_in_portal but no download, the portal must:

not show download buttons

prevent direct file URL access

block right-click save, print (as much as possible)

use time-limited signed streaming

watermark + audit log all views

4) Core Features
4.1 Agreement Management

Each Agreement record must capture:

Agreement Info

University/Provider Name (linked to Institutions)

Agreement Title (e.g., “2026 Agency Agreement – Bangladesh”)

Agreement Type (Agency / Commission schedule / Addendum / Renewal)

Start Date

Expiry Date

Status (Draft / Active / Expired / Terminated / Renewal in progress)

Territory (Country-based)

Territory Country (dropdown)

Optional: Region, Sub-region, City (future)

Notes / Internal remarks (restricted field: only certain roles can see)

Targets (if any)

Targets can be configured as:

Target Type: None / Monthly / Intake-based / Yearly

Target Metric:

Applications submitted

Enrolments

Student starts

Revenue (optional)

Target Value (number)

Target Period:

Month (YYYY-MM)

Intake (T1/T2/T3 or custom intake)

Year (YYYY)

Target Notes

A single agreement can have multiple targets (e.g., yearly + intake).

4.2 Commission Structure (Very Flexible)

Commission can be:

Commission Mode

Percentage OR Flat Amount

Commission Frequency/Scope:

Per subject

Per trimester/term

First-year only

Full course

Per enrolled credit points (optional future)

Commission Rules Data Fields (configurable)

Commission Name/Label (e.g., “Standard UG Commission”)

Level: UG / PG / Diploma / ELICOS / Package / Any

Commission Type: Percentage or Flat

Percentage Value (if Percentage)

Flat Amount + Currency (if Flat)

Basis:

Per subject (requires “subject count logic”)

1st year

Full course

Per intake

Subject Count Rules (if “per subject”):

Example: “2 subjects per block, up to 8 subjects per year”

Max subjects payable per year (number)

Pay only on enrolled subjects vs completed subjects (toggle)

Conditions:

Payable only if student remains enrolled until census date (toggle)

Clawback conditions (text field)

Bonus rules (optional)

Effective Start/End dates (rule-level, optional)

Attach supporting notes

Must allow multiple commission rules per agreement.

4.3 Agreement Document Upload & Secure Viewer
Upload

Allowed file types: PDF only (recommended). (DOCX allowed but convert to PDF server-side for viewing)

Max file size: set (e.g., 25–50MB)

Store file encrypted at rest

Store version history:

v1, v2, v3 with upload date, uploaded by, reason/note

View in Portal (NO DOWNLOAD)

For users without document.download:

Document opens in secure viewer embedded in portal

Must use:

Streamed delivery (chunked) with short-lived token

Watermark overlay: user name + email + timestamp + “CONFIDENTIAL”

Disable:

download button

direct file URL access

browser caching as much as possible

View events logged:

opened, page count/time (basic), closed, IP/device info

Note: “No download” can’t be 100% guaranteed (screenshots exist), but implement strong controls + auditing.

4.4 University Agreement Contacts (Multiple)

Each agreement (or university) can store multiple contacts for renewal and updates:

Required fields:

Full Name

Position/Title

Contact Number

Email Address

Country (dropdown)

Optional: Department

Optional: Notes

Is Primary Contact (boolean)

4.5 Expiry Reminder Emails

System must auto-send reminders when agreement is nearing expiry.

Reminder schedule (configurable)

Default:

90 days before expiry

60 days before expiry

30 days before expiry

14 days before expiry

7 days before expiry

1 day before expiry

On expiry date

Recipients (always)

au@studyinfocentre.com

info@studyinfocentre.com

partners@studyinfocentre.com

Optional recipients

Agreement Admins assigned to that territory

University contact(s) (only if you want external emails—default OFF)

Email content must include

University name

Agreement title

Territory country

Expiry date (absolute date)

Days remaining

Link to open agreement inside portal

Primary contact list summary

5) High Security Requirements (Critical)
5.1 Login security (verify before every login)

Mandatory MFA for all users:

TOTP (Google Authenticator) OR Email OTP

Prefer TOTP for stronger security

Device/session policy:

Short session lifetime (e.g., 30–60 minutes idle)

Force re-authentication for sensitive actions:

document view

document download

role changes

Account security:

Rate limiting

IP/device logging

Lockout after failed attempts

Password policy:

minimum length + complexity

breach detection (optional)

Admin actions require MFA confirmation

5.2 Data security

Encrypt:

Documents at rest (AES-256)

Sensitive columns (optional)

Strict audit logs:

login/logout

view agreement

edit agreement

upload/replace document

view document

download document

permission changes

5.3 Privacy & access control

Territory-based access option:

User can be limited to certain countries only

“Need-to-know” fields:

Some notes visible only to Admin/Compliance

10) Non-Functional Requirements

Performance: list pages must load fast with pagination

Logging: full security logs retained (e.g., 1–3 years)

Backups: daily backups (DB + docs)

Environment separation: dev/staging/prod

Monitoring: alert on failed email reminders / job failures

11) Acceptance Criteria (Must Pass)

Viewer can open agreement details but cannot edit.

Viewer can view document in portal but cannot download and cannot access file by direct URL.

Admin can upload a new document version and older versions remain accessible (based on permission).

Agreement expiry reminders are sent on schedule to the 3 emails.

All document views and downloads are logged in audit logs.

MFA is required at login for all users.

Permission changes are logged and require admin MFA confirmation.
