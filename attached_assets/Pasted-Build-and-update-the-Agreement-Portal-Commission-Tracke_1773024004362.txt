8. ALLOW MULTIPLE PROVIDERS / MULTIPLE ENROLMENTS FOR SAME STUDENT
====================================

The system must support one student enrolling with multiple providers and/or multiple courses.

Business cases:
- One student may enroll in 2 providers at the same time
  Example:
  - Diploma at one college
  - Bachelor at another college
- One student may do:
  - EAP at one provider
  - Bachelor/Master at another provider
- Provider may be same or different depending on the course structure

So:
- Do NOT block all repeated student names globally
- Do NOT block all repeated Agentsic IDs globally
- Allow the same student to exist in multiple valid enrolment records where duplicate rules are not violated

Required data model approach:
Create student-enrolment style records rather than assuming one student can only have one provider/course.

Recommended structure:
- Student master profile
- Enrolment records linked to student
- Each enrolment contains:
  - Provider
  - Intake Year
  - Intake Term
  - Course Level
  - Course Name
  - Duration
  - Commission details
  - Status
  - Notes

This way one student can have:
- Multiple providers
- Multiple courses
- Same provider with multiple valid course/enrolment rows if not duplicate by blocked rules

====================================

Also allow multiple course details under the same provider for the same student where valid.

Examples:
- Same provider, different course/package

Required behavior:
- System should support one student having multiple enrolment/course records even under same provider when those represent different valid course details
- Duplicate check should not wrongly block valid multiple-course cases
- To support this properly, include course-level detail in enrolment records

Important implementation note:
Duplicate blocking must follow the exact rules above, not broad assumptions.

10. RECOMMENDED DUPLICATE LOGIC DESIGN
====================================

To avoid blocking valid multi-provider or multi-course records, use enrolment-level uniqueness checks.

Recommended logic:
1. Student profile can exist once or be reused
2. Each provider/course/intake combination becomes an enrolment row
3. Duplicate validation runs before insert/update

Suggested backend validation flow:
- Normalize values before comparing:
  - trim spaces
  - lowercase for comparison
  - remove duplicate internal spacing where relevant
- Check duplicate rules in order:
  1. Student Name + Agentsic ID
  2. Student Name + Provider + Student ID
  3. Provider + Student ID
- Return exact rule failure

Important:
Because valid multi-provider and multi-course cases exist, design database carefully so real enrolments are stored separately from summary student profile.
