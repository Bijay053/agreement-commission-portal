1) Production-ready Database Schema (PostgreSQL)

Assumption: Django + DRF + PostgreSQL. This schema is normalized, supports versioned documents, flexible commission rules/targets, per-country scoping, and strong auditing.

countries

id (PK, smallint)

iso2 (varchar(2), unique, not null)

name (varchar(128), not null)

universities

id (PK, bigserial)

name (varchar(255), unique-ish; create unique index on lower(name))

country_id (FK -> countries.id, nullable)

website (varchar(255), nullable)

created_at, updated_at (timestamptz)

Indexes

idx_universities_name_lower on lower(name)

id (PK, bigserial)

(enum via check: ‘agency’, ‘commission_schedule’, ‘addendum’, ‘renewal’, ‘mou’, ‘other’)

2) Security Design: “View Document but Cannot Download”

Important truth: you cannot guarantee 100% prevention (screenshots/phone cameras exist). The goal is:

prevent easy download & direct link access

minimize leakage

strong deterrence (watermark)

traceability (audit logs)

Below is a proven approach that works well in production.

2.1 Threat Model (What we must stop)

✅ Must stop:

“Download” button and “Save as”

Copying a direct file link and opening it outside portal

Sharing a long-lived URL

Fetching the PDF from browser cache easily

Unauthorized users accessing documents

⚠️ Cannot fully stop (but deter + track):

Screenshots / screen recordings

Photos from another device

2.2 Secure Architecture (Recommended)
A) Store documents privately in object storage

S3 bucket private (or equivalent)

No public ACL

Access only from backend service role

B) Never serve raw document URLs directly

Instead:

User requests document view: POST /documents/{id}/view-token

Backend checks:

user authenticated + MFA session valid

has document.view_in_portal

country access allowed

document belongs to agreement user can view

Backend generates short-lived signed token (JWT) for viewing:

expires in 2–5 minutes

includes user_id, doc_id, permitted_action=view, nonce

Frontend loads secure viewer using this token.

C) Stream pages (not whole PDF download)

Best practice:

Convert PDF into page images (tiles) or per-page rendering server-side.

Serve as image streams or chunked responses.

This prevents a single “download PDF” network call.

Two common patterns:

Pattern 1 (Strongest): Pre-render PDF pages to images

On upload: backend renders PDF pages into PNG/WebP images stored privately

Viewer loads:

GET /documents/{id}/pages/{pageNo}?token=...

Pros: very hard to reconstruct a clean PDF

Cons: extra storage + processing, but worth it for “high security”

Pattern 2 (Medium): PDF streaming proxy with range requests blocked

Backend proxies PDF bytes but:

uses signed URL short TTL

disables Content-Disposition: attachment

sets Cache-Control: no-store

Pros: easier

Cons: user can still capture full PDF from network tools if skilled

Recommendation: Pattern 1 for your “very sensitive” requirement.

2.3 Watermarking (Deterrence)
Visible watermark overlay (frontend)

Always show watermark layer on top of pages:

{user_full_name} | {user_email} | {timestamp} | CONFIDENTIAL

repeated diagonally

Burned-in watermark (backend, stronger)

If using Pattern 2 (PDF streaming), apply watermark into the PDF pages server-side for each view session (expensive).
If using Pattern 1 (images), watermark can be burned into images during rendering (best).

2.4 Disable Download / Print (UI Controls)

Frontend viewer should:

Hide download/print buttons

Block right-click save (basic)

Disable text selection (optional)

Fullscreen viewer support

Detect devtools? (optional; not reliable)

But enforcement must be backend, not UI.

2.5 Prevent Direct Access, Sharing, and Caching
Token rules

Token tied to:

user_id

doc_id

device/session_id (optional)

Token expires quickly (2–5 min)

One-time nonce stored in Redis to prevent reuse (optional but strong)

Headers

For all viewer endpoints:

Cache-Control: no-store, no-cache, must-revalidate, private

Pragma: no-cache

Expires: 0

X-Content-Type-Options: nosniff

Content-Security-Policy locked down

Referrer-Policy: no-referrer

X-Frame-Options: DENY (or allow same-origin if embedded)

Block “attachment”

Always serve with Content-Disposition: inline; filename="..." (never attachment)

For images: normal image response

2.6 Re-authentication for Sensitive View

Even after login:

Require recent MFA to view documents:

Example: if MFA verification older than 10 minutes, prompt OTP again before document opens.
This matches your “verify before login every time” intent, and also protects sensitive actions.

2.7 Logging & Audit (Very Important)

Log every sensitive action:

DOC_VIEW_START (user, doc, agreement, timestamp, ip/device)

DOC_VIEW_PAGE (optional aggregate: pages viewed + duration)

DOC_VIEW_END

DOC_DOWNLOAD (if allowed)

TOKEN_ISSUED (doc view token)

Include:

IP address

user agent

country (optional)

session id

Add alert rules:

Too many doc views quickly

Access from new country / suspicious IP

Multiple failed MFA attempts

2.8 Optional “Highest Security” Add-ons (If you want)

Device binding (register trusted devices)

IP allowlist for admin roles

Session concurrency limits (1–2 active sessions max)

DLP: block copy/paste; not fully reliable in browser

Signed viewer player (custom viewer that never requests whole PDF)

3) Implementation Notes (React + Python)
Backend (Django)

Store documents in S3 private bucket

On upload:

validate PDF

compute sha256

render pages -> store /docs/{doc_id}/pages/{n}.webp

store page count in DB (add page_count column in agreement_documents)

Endpoint:

POST /documents/{id}/view-token returns token + page_count

GET /documents/{id}/pages/{n} streams a single image if token valid

Enforce:

permission checks

country access checks

token expiry

no-store headers

Frontend (React)

Viewer loads page images sequentially (lazy load)

Watermark overlay always on

No download controls unless permission exists

Re-auth modal if required
